Flusso di progetto

Da zero a produzione — con ogni feature attivata al momento giusto

Ogni feature del template ha un gate: si attiva solo quando il contesto del progetto lo rende pertinente, mai di default. Questo flusso mostra quando e perche' ciascuna si accende su un progetto reale.

1
Giorno 0 — 30 minuti
Inizializzazione del progetto
Si incolla PROMPT-nuovo-progetto.md in Claude Code nella radice del progetto. La skill init-project-system esegue l'anatomia minima: crea .claude/ con memory/, context/ e rules/, istanzia le schede tecniche vuote, popola CLAUDE.md con la procedura di ripresa. Poi legge PACKAGES.md e propone i pacchetti pertinenti uno per uno.

Gate accettati in questo progetto: github-mcp (il progetto vive su GitHub), context7 (si usa Express e Drizzle in evoluzione rapida), playwright-mcp (test E2E previsti), ccusage (baseline misurazione token, sempre), vercel-mcp (deploy su Vercel). Gate rifiutati: LaTeX, knowledge-wiki (non necessari ora).
coreinit-project-system corePROMPT-nuovo-progetto opt-ingithub-mcp opt-incontext7 opt-inplaywright-mcp opt-inccusage opt-invercel-mcp
2
Giorno 1 — Prima sessione
Architettura e schede tecniche
All'avvio della prima sessione si legge memory/index.md (stato iniziale), si invoca sync-context per misurare la divergenza. Si popola STACK.md con stack, flussi e ruolo dei file; design-and-security.md con i paradigmi scelti.

security-auditor viene chiamato sullo schema di autenticazione JWT appena abbozzato: trova un finding HIGH (token senza scadenza configurabile) e uno MEDIUM (mancanza di rate limiting). Si corregge prima di andare avanti. La regola security-permissions.md e' gia' attiva: qualsiasi tentativo di leggere .env viene bloccato dalle deny rules.
coresync-context corememory/index.md coreSTACK.md + design-and-security.md agentesecurity-auditor regolasecurity-permissions opt-incontext7 (docs Express 5)
3
Settimana 1 — Sviluppo attivo
Loop di sviluppo quotidiano
Ogni sessione inizia leggendo memory/index.md e current-work.md, poi sync-context misura il drift. Solo le schede pertinenti alla feature attiva vengono lette. Il RESUME-PROMPT.md (in _notes/, ignorato) viene aggiornato a fine sessione.

github-mcp gestisce issue e PR senza uscire da Claude Code. code-reviewer viene invocato prima di ogni PR merge: classifica i finding per severita' (CRITICAL blocca il merge). context7 risolve le ambiguita' sulla documentazione di Drizzle ORM.

ccusage mostra la baseline: media 45K token per sessione. Si decide di attivare ponytail per tenere il codice piu' compatto.
coresync-context (ogni sessione) agentecode-reviewer (pre-merge) opt-ingithub-mcp opt-incontext7 opt-inccusage opt-inponytail
4
Settimana 1 — Sessione lunga (6 ore)
Igiene di sessione e token economy
A meta' sessione il contesto raggiunge il 52%. Si lancia /compact con istruzione guidata: "focus on the auth flow decisions and the Drizzle schema we finalized". La qualita' delle risposte resta alta.

Prima di chiudere, si fa scrivere a Claude un HANDOFF.md con le tre decisioni prese, i quattro file toccati e il prossimo passo concreto. La sessione successiva riparte da quel file invece che da zero.

L'output dei test vitest e' verboso: si attiva rtk per comprimere l'output shell prima che entri nel contesto.
regolatoken-economy (/compact 52%) coreHANDOFF.md (pre-chiusura) opt-inrtk (output vitest) opt-inponytail
5
Settimana 2 — Testing e deploy
Test E2E, audit di sicurezza e deploy
playwright-mcp pilota il browser in locale per verificare le rotte API autenticate: nessun browser aggiuntivo, riusa Chromium gia' installato.

security-auditor esegue un full pass prima del deploy: trova un finding CRITICAL (path traversal su un endpoint di upload) che viene risolto prima del push. Il finding viene registrato in memory/decisions.md come decisione architetturale.

vercel-mcp monitora il deploy preview: differenza di latenza tra preview e produzione visibile direttamente in Claude Code. La variabile d'ambiente mancante viene aggiunta tramite vercel-mcp senza uscire dalla sessione.
opt-inplaywright-mcp agentesecurity-auditor (full pass) agentecode-reviewer (test review) opt-invercel-mcp corememory/decisions.md
6
Settimana 3-4 — Progetto in evoluzione
Memoria semantica e grafo di conoscenza
Dopo tre settimane memory/progress.md e' cresciuto a 400 righe. Si attiva claude-mem: memoria semantica persistente SQLite+Chroma, interamente locale. I due sistemi sono complementari: progress.md per la traccia versionata; claude-mem per il retrieval semantico cross-sessione.

graphify genera una mappa interattiva delle relazioni tra moduli: emerge che il modulo di autenticazione e' un nodo con 12 connessioni in uscita, molto piu' centrale del previsto. Cambia la priorita' del refactoring.

Si attiva codebase-memory-mcp perche' il codebase ha superato 50K righe: le query strutturali costano ora 200 token invece di 8K.
opt-inclaude-mem opt-ingraphify opt-incodebase-memory-mcp corememory/progress.md
7
Tag v1.0 — Documentazione finale
Onboarding, stile e riconciliazione finale
Prima del tag, sync-context passa in rassegna tutte le schede: tre sono marcate "da aggiornare" perche' il refactoring di settimana 3 ha spostato file coperti dalle paths. Si aggiornano solo quelle tre.

Il README pubblico viene scritto da Claude e poi passato a humanizer: rimuove 11 occorrenze di pattern AI. La regola interaction-style era gia' attiva sulla documentazione interna; humanizer serve per la prosa esterna.

La skill onboard genera la guida per il nuovo collaboratore leggendo le schede, la storia dei commit e le decisioni architetturali.
coresync-context (finale) skillonboard opt-inhumanizer regolainteraction-style regolagit-commands-format
Cosa non hai dovuto fare in questo flusso: rileggere la documentazione da zero a ogni sessione, copiare stack trace in chat, aprire GitHub nel browser per gestire PR, cercare documentazione di Drizzle ORM su MDN, ricordare a mano le decisioni architetturali di tre settimane fa, scrivere il README senza aiuto stilistico.
